# Graph API 転送最適化制御 設計書 v1.0

## 1. 目的

本設計は、APIスロットリング（429）を回避しつつ、最大スループットで安定したデータ転送を実現するための制御アルゴリズムを定義する。

---

## 2. 設計方針

* 429を完全にゼロにはしない（最適点は境界付近）
* 「並列数」ではなく「リクエスト流量（レート）」を主制御とする
* 短期・中期の時間窓で状態を評価する
* インフライト（未完了リクエスト）を負債として扱う
* フィードバック制御により自己最適化する

---

## 3. 全体アーキテクチャ

```
Queue
  ↓
Dispatcher（レート制御）
  ↓
Worker（並列上限付き）
  ↓
Graph API
  ↓
ログ収集 → リアルタイム分析 → 制御フィードバック
```

---

## 4. 制御要素

### 4.1 レート制御（主制御）

* 単位：requests/sec
* Queueからの取り出し速度を制御

```
dispatch_rate = f(429_rate_30s)
```

---

### 4.2 並列制御（補助）

* 上限としてのみ使用
* 主制御ではない

```
max_concurrency = 固定値（例: 16）
```

---

### 4.3 インフライト制御（重要）

#### 定義

```
in_flight = 実行中リクエスト数 + Retry待ちリクエスト数
```

#### 制御

```
if in_flight > threshold:
    dispatch停止
```

---

## 5. 時間窓設計

### 短期ウィンドウ（5秒）

* スパイク検知
* 緊急制御用

### 中期ウィンドウ（30秒）

* 安定判断
* レート調整

---

## 6. 429制御ロジック

### 6.1 可変減衰

```
factor = clamp(1 - k * 429_rate, 0.3, 0.9)
```

適用：

```
rate *= factor
```

---

### 6.2 ヒステリシス制御

```
if 429_rate_5s > 10%:
    緊急減速

elif 429_rate_30s > 3%:
    緩やかに減速

elif 429_rate_30s == 0:
    加速
```

---

### 6.3 加速制御

```
if 安定状態:
    rate += increment
```

または

```
rate *= 1.05
```

---

## 7. Retry-After対応

* Retry-Afterは最優先で遵守
* 対象リクエストはin_flightに含める

```
on 429:
    wait(retry_after)
```

---

## 8. パルス制御（オプション）

### 目的

* 内部カウンタのリセット
* 長期スロットリング回避

### モデル

```
cycle = 5秒
duty = 0.8

80%: 通常送信
20%: 送信停止
```

### 適応

```
429増加 → duty低下
安定 → duty上昇
```

---

## 9. ログ設計

### 必須項目

* timestamp
* status_code
* latency
* endpoint
* file_size / chunk_size
* concurrency
* rate

### 集計指標

* 429率（5秒 / 30秒）
* RPS
* 成功率
* 平均レイテンシ

---

## 10. スコア関数（最適化指標）

```
score = throughput - penalty(429) - latency_penalty
```

---

## 11. 制御ループ

周期：1秒

```
1. ログ集計更新
2. 429率算出
3. レート調整
4. インフライトチェック
5. dispatch実行
```

---

## 12. 期待効果

* 429の局所化（スパイク抑制）
* スループットの安定化
* 過剰待機の削減
* 自動的な最適点収束

---

## 13. 拡張ポイント

* ファイルサイズ別レーン分離
* ユーザー単位制御
* 予測型制御（事前減速）
* 軽量強化学習（バンディット）

---

## 14. 結論

本設計は以下を実現する：

* レート主導制御
* インフライト吸収
* 多時間窓フィードバック

これにより、APIスロットリング環境下でも高効率・高安定な転送を可能とする。
